## Что такое Xynapse IDE

Xynapse IDE — это десктопная среда разработки на базе Electron/TypeScript, построенная на ядре VS Code и расширенная мощной AI-системой собственной разработки. В отличие от AI-плагинов, которые существуют отдельно от редактора, Xynapse интегрирует интеллект непосредственно в рабочий процесс: ассистент понимает контекст всего проекта, а не отдельных файлов.

**Три кита системы:**

| Подсистема | Что делает |
|---|---|
| **Xynapse Assistant** | Диалоговый AI-ассистент с полным контекстом проекта, инлайн-редактированием и slash-командами |
| **Xynapse Autocomplete** | Инлайн-автодополнение в реальном времени, работающее независимо от чат-модели |
| **Xynapse Council** | Мультиагентная система планирования — несколько AI-экспертов обсуждают задачу и синтезируют план |

## Xynapse Assistant

Встроенный AI-ассистент живёт в боковой панели и работает как полноценный соавтор кода.

### Режимы работы

**Чат** — задавайте вопросы на естественном языке. Ассистент видит:
- Открытые файлы и выделенные фрагменты
- Структуру всего проекта
- Вывод терминала и список ошибок
- Git diff текущих изменений

**Редактирование (Edit & Apply)** — выделите любой фрагмент кода, опишите что нужно изменить. Ассистент сгенерирует diff и применит его прямо в редакторе одним нажатием. Никаких копипастов.

**Slash-команды** — быстрые действия прямо из поля ввода:

| Команда | Действие |
|---|---|
| `/edit` | Редактировать выделенный код |
| `/explain` | Объяснить код |
| `/comment` | Добавить комментарии |
| `/test` | Сгенерировать тесты |
| `/council` | Запустить мультиагентное планирование |

**Контекстное меню** — кликните правой кнопкой на любом фрагменте кода и получите AI-действия: комментирование, документирование, исправление ошибок, оптимизация, исправление грамматики.

### Система ролей моделей

Xynapse не ограничивает вас одной моделью на всё. Каждая задача может использовать отдельную модель, оптимальную именно для неё:

| Роль | Назначение |
|---|---|
| `chat` | Основной диалог и вопросы |
| `edit` | Редактирование кода (инлайн-изменения) |
| `apply` | Применение предложенных изменений |
| `autocomplete` | Инлайн-автодополнение |
| `summarize` | Суммаризация длинных контекстов |
| `embed` | Векторное представление для поиска по кодовой базе |
| `rerank` | Переранжирование результатов поиска |

Это позволяет, например, использовать быструю лёгкую модель для автодополнения и мощную для чата — без компромиссов между скоростью и качеством.

## Xynapse Autocomplete

Система дописывания строк — инлайн-автодополнение работает параллельно с ассистентом и не мешает диалогу.

- **Реальное время** — подсказки появляются по мере набора, без видимой задержки
- **Отдельная модель** — автодополнение использует собственную модель с ролью `autocomplete`, независимо от чат-модели
- **Настраиваемый триггер** — контролируйте задержку, минимальную длину запроса и условия отображения
- **Приём Tab** — принимайте предложение целиком нажатием Tab

## Зашифрованный экспорт конфигурации

Xynapse использует локальные аккаунты — без облачной авторизации. Для переноса конфигурации между ПК:

- **Экспорт** (`Ctrl+Shift+P` → "Xynapse: Export Encrypted Config Backup") — собирает `config.yaml`, `config.json`, `profile.json` в один `.enc` файл, зашифрованный AES-256-GCM + PBKDF2
- **Импорт** (`Ctrl+Shift+P` → "Xynapse: Import Encrypted Config Backup") — расшифровывает и восстанавливает конфигурацию
- **Git-синхронизация** — `.enc` файл можно хранить в любом git-репозитории для синхронизации между машинами

## Xynapse Council

<div align="center">
<img src="./Pics/Council.png" alt="Xynapse Council" width="600"/>
</div>

Xynapse Council — это система мультиагентного планирования, где задача не просто решается одной моделью, а обсуждается командой специализированных AI-экспертов. Результат — взвешенный план с явно разрешёнными конфликтами.

### Алгоритм: 3-фазная дискуссия

**Фаза 1: Независимый анализ**

Каждый агент получает задачу и анализирует её самостоятельно, не видя ответов других. Это устраняет эффект «первого слова» — ни один агент не влияет на мнение остальных на начальном этапе.

Формат ответа каждого агента:
```
## Предложение      — конкретное решение задачи
## Риски            — потенциальные проблемы и узкие места
## Ключевые решения — технические решения, требующие обсуждения
```

**Фаза 2: Перекрёстная критика** (0–2 раунда в зависимости от сложности)

Каждый агент видит ВСЕ ответы из предыдущих фаз и обязан критически их оценить. Агент не просто соглашается или не соглашается — он предлагает конкретные изменения.

Формат ответа:
```
## Согласен         — что в других предложениях верно
## Не согласен      — конкретные возражения с обоснованием
## Предлагаю изменить — конкретные альтернативы
```

**Фаза 3: Синтез плана**

Планировщик получает полную историю дискуссии и синтезирует итоговый план, явно разрешая все конфликты между агентами.

```
## Спорные решения  — список конфликтов и как они разрешены
## Итоговый план    — пошаговый план реализации
## Структура файлов — конкретные файлы и их назначение
```

### Количество LLM-вызовов в зависимости от сложности

| Сложность | Фаза 1 | Фаза 2 (раунды критики) | Всего вызовов |
|-----------|--------|--------------------------|----------------|
| Легко | N агентов | 0 раундов | N + 1 |
| Средне | N агентов | 1 раунд | 2N + 1 |
| Сложно | N агентов | 2 раунда | 3N + 1 |

При 4 агентах и сложном уровне: **13 LLM-вызовов** с полной трассировкой дискуссии.

### Контекст проекта

Перед началом дискуссии Council автоматически собирает контекст проекта:
- **Файловое дерево** (глубина 2, фильтрация по `.gitignore`)
- **README** (до 2000 символов)
- **Файлы зависимостей** (`package.json`, `Cargo.toml`, `requirements.txt`, и др., до 3000 символов)

Контекст инжектируется в системное сообщение каждого агента — они знают, с каким проектом работают.

### Настраиваемые роли агентов

По умолчанию Council использует 4 роли:

| Роль | Специализация |
|---|---|
| **Architect** | Системный дизайн, структура проекта, технологический стек |
| **Developer** | Конкретная реализация, алгоритмы, паттерны кода |
| **Reviewer** | Качество кода, безопасность, производительность, best practices |
| **Tester** | Тестируемость, граничные случаи, интеграционные сценарии |

Роли полностью настраиваемы: можно создать любое количество агентов с произвольными специализациями.

### Назначение моделей на роли

Каждый агент Council может использовать отдельную AI-модель. Например:
- Architect → GPT-4o (архитектурное мышление)
- Developer → Claude Sonnet (реализация)
- Reviewer → Gemini (анализ кода)
- Tester → YandexGPT (тестирование)

### Выходные файлы

- **`council-plan.md`** — финальный план реализации (открывается автоматически)
- **`council-discussion.md`** — полный лог дискуссии по фазам (опционально)

### Запуск

```
/council Реализовать систему аутентификации с JWT и refresh токенами
```

Или кнопка Council в панели ввода ассистента.

## Архитектура

### Поток данных: конфиг → бэкенд → интерфейс

```
~/.xynapse/config.yaml
    → loadYaml.ts      — парсинг YAML, развёртывание ролей
    → models.ts        — создание экземпляров BaseLLM по конфигу
    → load.ts          — сериализация по ролям для фронтенда
    → core.ts          — передача через postMessage в webview
    → ParallelListeners.tsx — приём на стороне GUI
    → Redux store      — состояние конфигурации
    → ModelSelect.tsx  — отображение списка моделей
```

### Слои системы

```
┌─────────────────────────────────────────┐
│              VS Code Editor             │  ← Electron/TypeScript ядро
├─────────────────────────────────────────┤
│         Extension Host Process          │  ← Node.js процесс расширения
│   XynapseGUIWebviewViewProvider         │
│   XynapseConsoleWebviewViewProvider     │
├─────────────────────────────────────────┤
│            Xynapse Core                 │  ← Бэкенд ассистента
│   LLM Router → BaseLLM providers        │
│   Config YAML → Model role mapping      │
│   Slash commands (council, edit, ...)   │
│   Context providers (code, diff, ...)   │
├─────────────────────────────────────────┤
│            Xynapse GUI                  │  ← React/Vite webview
│   Chat interface → Redux state          │
│   Model selector → Role-based display   │
│   Council dialog → Phase streaming      │
└─────────────────────────────────────────┘
```

## Конфигурация

Ассистент настраивается через `~/.xynapse/config.yaml`. Полный пример:

```yaml
name: "Мой конфиг"
version: "1.0.0"

models:
  # Основная модель для чата
  - name: "GPT-4o"
    provider: "openai"
    model: "gpt-4o"
    apiKey: "sk-..."
    roles:
      - chat
      - edit
      - apply

  # Быстрая модель для автодополнения
  - name: "YandexGPT Lite"
    provider: "yandex_gpt"
    model: "yandexgpt-lite"
    apiKey: "AQVN..."
    roles:
      - autocomplete
    requestOptions:
      extraBodyProperties:
        folderId: "your-folder-id"

  # Мощная модель для суммаризации больших контекстов
  - name: "Claude Sonnet"
    provider: "anthropic"
    model: "claude-sonnet-4-6"
    apiKey: "sk-ant-..."
    roles:
      - summarize

rules:
  - "Ты — Xynapse Assistant, интеллектуальный помощник разработчика."
  - "При создании файлов всегда используй путь относительно корня проекта."

context:
  - provider: "code"
  - provider: "diff"
  - provider: "terminal"
  - provider: "problems"
  - provider: "codebase"
```

Шаблон конфигурации: `vscode/extensions/xynapse-assistant/xynapse-config.yaml`

## Быстрый старт

### Требования

- Node.js 20.19.0+ (LTS)
- Python 3.10+
- Windows 10/11

### Запуск через лаунчер

```powershell
git clone https://github.com/jabrailkhalil/Xynapse.git
cd Xynapse
xynapse.bat
```

Лаунчер предоставляет интерактивное меню: сборка, запуск, dev-режим, watch-режимы, релизная упаковка.

### Ручная сборка

```powershell
cd vscode
npm install
npm run compile
.\scripts\code.bat
```

### Релизная сборка (Win64)

```powershell
cd vscode
npm run gulp vscode-win32-x64
# Результат: .build/VSCode-win32-x64/
```
